Staffing Intelligence

Real-time resource planning dashboard for a ~25-person NetSuite consulting practice. Multi-tenant, role-based, AI-powered.

Architecture
System Overview
Four-tier architecture: browser client, Express API server, Supabase (Postgres + Auth), and Claude API for AI features. Hosted on Railway with auto-deploy from GitHub.
The server acts as a gateway between the browser and all backend services. No client ever talks to Supabase or Claude directly — all requests route through Express, which enforces auth, role checks, and tenant isolation before touching data.

Access control
Authentication & RBAC
Five roles with distinct permission boundaries. Auth flows through Supabase with server-side session persistence.

Auth flow

1. User submits email + password to POST /api/auth/login
2. Server calls supabase.auth.signInWithPassword()
3. On success, extracts role and tenant_id from JWT app_metadata
4. Stores in express-session: userId, email, role, tenant_id, userToken
5. All subsequent requests use session — no JWT passed from client
6. requireAuth middleware checks session on every /api/* route
7. requireRole(...roles) middleware gates sensitive operations

Isolation
Multi-Tenant Architecture
Complete data isolation between tenants at the database layer. No tenant can ever see another tenant's data.

How tenant isolation works

1
JWT contains tenant_id
When a user logs in, app_metadata.tenant_id is extracted from the JWT and stored in the server session. Every request carries this context.
2
tId(req) on every query
const tId = req => req.session?.tenant_id || process.env.TENANT_ID — every Supabase read/write passes this to filter by tenant. No route handler ever reads data without tenant scoping.
3
RLS as safety net
Row Level Security policies on every table filter by tenant_id = auth.jwt() ->> 'tenant_id'. Even if application code has a bug, the database blocks cross-tenant reads. serviceClient bypasses RLS but always passes tenant_id explicitly.

Tab routing

Five tabs: Overview (Ctrl+1), Resource Allocation (Ctrl+2), Open Needs (Ctrl+3), Ask Claude (Ctrl+4), Settings (Ctrl+5). navigateTo(tabName) hides all panels, shows the target, and triggers a data refresh. Sidebar nav items, keyboard shortcuts, and command palette all call the same function.

Inline editing pattern (heatmap)

Click a heatmap cell → input appears → type hours → blur or Enter saves via POST /api/save-staffing. Changes are tracked in _pendingStaffing Map until confirmed. Enter key navigates to next cell with a polling pattern (setInterval 50ms, 20 attempts) for post-render DOM queries. Quick Fill always uses the API path — DOM fast path was removed to avoid ambiguity with multiple project sub-rows.
Intelligence
AI Integration
Three AI-powered features built on Claude API (claude-sonnet-4-6). All context is built server-side from live tenant data — Claude never accesses the database directly.
💬
Ask Claude
NL Query
Natural language questions over staffing data. Server builds a context string with: 12-week window, all consultants (name, level, skills, hours), all projects, all open needs. Claude responds conversationally. Markdown rendered via marked.js.
🎯
Recommendations
Matching
AI-matched consultants for open needs. Matches on: level + allSkillSets.includes() (any-match, not primary-only). Availability = average available hours within need's date window. Sorted by availableHours desc. Badge: green ≥100%, yellow 50-99%, coral <50%. Cached per-tenant (60s TTL).
✨
Suggested Questions
Contextual
Claude generates 4-6 relevant questions based on the current data state (e.g. "Who's available next month?", "Which projects are understaffed?"). Refreshes on each Ask Claude tab load. Provides a zero-effort entry point for users unfamiliar with the data.

Real-time
Server-Sent Events (SSE)
Real-time updates pushed from server to all connected clients. Used for cross-user data synchronization.
Endpoint: GET /api/events

Clients connect via EventSource. Server maintains a list of active connections. When any mutation occurs (consultant updated, assignment saved, need closed), the server calls broadcast({ type: 'consultant-updated', ... }) which pushes the event to all connected clients. Clients refresh their local data on receiving relevant events.

Event types: consultant-updated, assignment-saved, need-created, need-closed, need-updated

1. Push to main branch
2. Railway detects push, builds Node.js app

📦
Cache busting
app.js?v=N and styles.css?v=N query strings in index.html. Incremented on every deploy with frontend changes. Prevents stale browser caches.

Database migrations

Run via Supabase SQL Editor (paste raw SQL). No migration framework. The Supabase service key is an opaque API token for PostgREST — it cannot connect to Postgres directly. Do not attempt node -e or psql migrations.

Security
Security Model
Defense in depth: authentication, authorization, data isolation, and input validation at every layer.
LayerMechanismDetails
AuthenticationSupabase Auth + express-sessionJWT-based login, server-side session persistence. Sessions are in-memory (MemoryStore) — lost on restart, acceptable for single-instance.
AuthorizationrequireAuth + requireRole middlewareEvery /api/* route requires auth. Sensitive routes (writes, admin) require specific roles. Checked server-side before any data operation.
Tenant isolationRLS + tId(req)Database-level row policies filter by tenant_id. Application layer always passes tId(req) as a filter. Double enforcement — DB blocks even if app has a bug.
Service key isolationServer-only serviceClientSUPABASE_SERVICE_KEY never sent to browser. Only server.js and supabaseReader.js use it. All client requests go through Express middleware first.
Input validationRoute-level checksRequired fields validated before DB writes. Level, hours, dates checked for type and range. SQL injection prevented by Supabase client parameterization.
XSS preventionNo raw HTML injectionUser input rendered via textContent or escaped. marked.js sanitizes markdown in Ask Claude responses.
Password handlingSupabase AuthPasswords hashed by Supabase (bcrypt). Server never sees or stores plaintext passwords beyond the login request.
